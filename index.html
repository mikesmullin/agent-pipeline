A framework for building agents that run as a loop, not a script.

Borrow the game-engine loop and the ECS pattern. Keep your work items on disk as plain YAML, so the loop is crash-safe, resumable, and editable in your text editor while it runs. Confine the model to tiny, single-decision microagents; keep everything else honest deterministic code.

🚀 Quick start 🍪 Build a pipeline ⚙️ For contributors

Most "AI agent" code is a one-shot prompt chain: it runs top to bottom, and when it crashes halfway through a thousand items, you start over and hope. Agent Pipeline is the opposite.

The hard part of a real pipeline isn't calling the model — it's everything around it: where state lives, which step runs next, who's allowed to write which field, and how you stop the model from quietly doing the wrong thing. This framework gives you those primitives and the guardrails, so you spend your time on business logic, not plumbing.

Why a loop, not a script #

The core idea

A script processes items start to finish. A loop advances many items a little at a time, forever — exactly how a game engine renders thousands of objects: one tick advances every object one step. That single shift is what buys you crash-safety, resumability, and live editability.

🛟 Crash-safe by construction

State is YAML on disk, one file per work item. Kill the process mid-run and it resumes every item exactly where it left off — because the next tick simply re-queries what still needs doing.

🚧 The model can't go rogue

Every component field declares who may write it. A violation is a hard error at runtime and a failed build under pipeline check. The LLM never gets to scribble outside its lane.

🎯 Judgment is isolated

LLM calls live only inside one-decision microagents — one question, one typed answer. The rest of your pipeline is ordinary, testable, deterministic code.

📐 Docs that stay true

pipeline docs regenerates the stage / schema / access tables from the source, so the reference can't drift from the code. pipeline check is the static linter that keeps the contract honest.

Why it matters. When you migrate a thousand dashboards (the project this framework was forged on), a one-shot script is hopeless: a flaky cluster, a memory balloon, or one bad item wedges the whole run. A loop shrugs those off — the bad item stays stuck and self-explains, every other item keeps flowing, and you fix the class of problem once.

Quick start #

From zero to a running loop in four commands

1 · Install

Built for Bun + CoffeeScript. Link it from local disk (not on npm yet):

# in the framework repo
cd pipeline
bun install
bun link            # registers the global `pipeline` command + a linkable dep

2 · Scaffold a project

pipeline new writes a complete, runnable, activity-first project so every concept is visible in one place. Everything is multi-activity — a single-activity project is just a one-item activity list, so the structure never changes when a second activity arrives.

# one activity named `notes`, with three stages:
pipeline new my-agent --activity notes --stages ingest,classify,publish
cd my-agent
bun link pipeline   # resolve `import … from 'pipeline'` to your local framework
bun install

The CLI uses ordered, repeatable scope flags: each --activity <id> opens a scope and the flags after it apply until the next --activity. Repeat to host several pipelines in one project:

pipeline new big \
  --activity kibana --stages ingest,resolve,publish,verify,report \
  --activity beam   --stages fetch,convert,publish

3 · Run it, then drop in work

bun agent.coffee                          # start the loop (ticks every few seconds)
echo "say hello" >> notes/db/_drop.md     # hand the 'notes' activity a unit of work

4 · Debug without running the whole loop — walk

pipeline walk runs a selection of entities through a selection of stages. A single entity prints its gate trace + component diff; many entities stream one at a time under a live progress meter.

pipeline walk --activity notes --entity <id> --stage classifySystem   # one entity, one stage
pipeline walk --activity notes --entities 0..49 --stages ingest..publish
pipeline walk --activity notes --once                                 # all entities × all stages, once

Walk is safe while the loop runs. It streams one entity at a time (bounded memory) and is the fastest way to exercise gate logic + a stage on real entities. Pass --force / --no-lock to bypass the cwd lock and walk while the loop is draining the corpus.

Build your first pipeline #

The scaffold, end to end

Each activity gets its own directory holding everything it needs. One system file per stage, already chained:

my-agent/
├─ agent.coffee              # the loop driver — one line: await runPipeline()
├─ config.yaml              # loop knobs (model, pipeline_width, interval)
├─ .code-review.yaml        # shared LLM convention rules
└─ notes/                   # ── one activity ──
   ├─ activity.yaml         #   manifest: stages + pipeline + agents
   ├─ schema.yaml           #   components + per-field ACL (the contract)
   ├─ config.yaml           #   per-activity domain config
   ├─ systems/              #   ingest.coffee · classify.coffee · publish.coffee
   ├─ microagents/          #   the one-decision LLM steps
   ├─ models/components/    #   one validated accessor per component
   ├─ docs/PIPELINE.md      #   GENERATED by `pipeline docs`
   └─ db/<id>.yaml          #   the entities (disk = source of truth)

Generate the next piece, already wired

You never hand-write the boilerplate — generators emit it wired to the schema and conventions:

pipeline g system    enrich     # a new stage
pipeline g component triage     # a new typed field-bag (+ its ACL stub)
pipeline g microagent classify  # a one-decision LLM step

Keep it honest as you go

pipeline check      # static schema/ACL linter — fails the build on a violation
pipeline docs       # regenerate the stage / schema / access reference from source
pipeline review     # run the bundled LLM convention rules (code-review)

The game-engine loop #

One tick advances everything a little

Your agent.coffee is one line:

import { runPipeline } from 'pipeline'
await runPipeline()

runPipeline() discovers every <id>/activity.yaml, loads its systems, and runs each activity's pipeline forever. Nothing else lives in the loop body — that is the whole standard.

Under the hood the loop is dual-mode. A project with activity.yaml manifests runs the multi-activity loop; a bare config.yaml systems: scaffold is synthesized into a one-activity set and runs the same path. There is exactly one execution model.

The mapping. Entities are dumb data · Components are data namespaces · Systems are the behavior · the loop is the frame tick. If you've ever written a game update loop, you already know this shape.

ECS & the five primitives #

Everything is one of five things

Agent Pipeline borrows the Entity-Component-System pattern game engines use to keep thousands of objects organized, and adds one primitive for agentic work — the agent, the only place an LLM exercises judgment.

① Entity

db/<id>.yaml

One unit of work in flight. Opaque except for its id — just an id plus whatever data has accreted onto it. Disk is the source of truth.

② Component

a top-level key in the YAML

A named, typed bag of fields (note, verify, …). Declared once in schema.yaml, which is also the access-control list.

③ System

systems/<stage>.coffee

A function that advances entities through one stage. It pulls the entities it wants (queries the world; never handed one), does deterministic work, and saves.

④ Agent / Microagent

microagents/<name>.coffee

The only place an LLM decides. A microagent is the focused subset: one question, one typed tool-call answer.

⑤ The loop

agent.coffee → runPipeline()

The engine tick. Framework-owned; runs each activity's systems every few seconds, with bounded per-stage concurrency.

Why ECS for agents? Because work items accrete capability exactly like game objects accrete components. An entity isn't "a dashboard at stage 4" — it's "an entity that has a fetch and a dataview but not yet a verify." Systems query by component presence (archetype style), so adding a stage is adding a component, not editing a state machine.

Entities — disk is the source of truth #

Crash-safety, made of plain files

An entity is a YAML file at <activity>/db/<id>.yaml. It is just an id plus the components that have accreted onto it. The World cache is a performance layer only — every read goes through the model and re-reads from disk when the file's mtime changed, so an operator editing a YAML in their editor takes effect on the next tick.

# db/a0f4a9.yaml — an entity mid-flight
id: a0f4a9
fetch:                  # component ① — written by the ingest system
  status: original
dataview:               # component ② — written by the resolve system
  resolved: true
  k8_dataview_id: 'a786…'
# no `verify:` component yet → the verify stage's gate will pick this up next

The stage lives inside the entity; files never move. Two sanctioned styles: a workflow._stage field, or a status field plus component-presence (e.g. fetch.status + whether dataview is written). Your editor tabs stay stable as work progresses — nothing is shuffled between folders.

The full Entity API is below; the headline is that you rarely touch it directly — you read and write through validated component accessors, and the framework owns the load/save/evict lifecycle.

Components & the access-control list #

The schema is the contract AND the ACL

A component is declared once in schema.yaml. Crucially, every field lists which subjects may write it — the access-control list the framework enforces at runtime and the linter enforces statically.

# schema.yaml
components:
  note:
    fields:
      text:
        type: string
        subjects: [system:notes/ingest]      # only ingest may write `text`
      label:
        type: string
        subjects: [agent:notes/classify]     # only the classify microagent may write `label`
      processed_at:
        type: ISO string
        subjects: [system:notes/classify]

Build a model with one line; it generates a validated getter and setter per field:

import { defineComponent } from 'pipeline'
export NoteComponent = defineComponent 'note', ['text', 'label', 'processed_at']

# every accessor takes the calling SUBJECT, then activityId, then id:
text = await NoteComponent.text SUBJECT, activityId, id            # getter (camelCase field)
await NoteComponent.setProcessedAt SUBJECT, activityId, id, isoNow  # setter (set + PascalCase)

A violation is fatal. If a subject writes a field it isn't allowed to, the SchemaValidator exits the process (exit 1) at runtime, and pipeline check fails the build before you ever run. This is the single mechanism that stops the model — or a careless refactor — from scribbling outside its lane.

For compound / multi-field writes or array components, hand-roll a Component subclass and call @check(subject, field) per field; declare the method → fields mapping in schema-aliases.yaml so the linter can trace it.

Systems — pull, work, save #

Behavior lives here, never in the model

A system advances entities through one stage. Its first act is to query the world for the entities it cares about — it is never handed one. It does deterministic work, then saves through validated accessors.

import { Entity, Gate } from 'pipeline'
import { NoteComponent } from '../models/components/note.coffee'

SYSTEM = 'system:notes/classify'
ACTIVITY = 'notes'
export GATE_FIELDS = ['note.text', 'note.processed_at']   # the fields the gate reads

export classifySystem = ->
  # PULL: select entities this stage should advance (the gate).
  targets = await Entity.query ACTIVITY, (e) ->
    return false unless Gate 'has text, not yet classified', (
      e.note?.text? and not e.note?.processed_at?)

  for entity in targets
    # … deterministic work (or call a microagent) …
    await NoteComponent.setProcessedAt SYSTEM, ACTIVITY, entity.id, new Date().toISOString()

Why pull, never push? Because a pushed entity couples the caller to the system's selection logic, and the system can no longer be reasoned about in isolation. When every system queries for its own archetype, the loop can run them in any order, resume after a crash, and re-pick an entity the instant a prior stage writes its component — no orchestration glue, no state machine to keep in sync.

A system may also export selectEligible + processOne (the worker-pool seams) so the framework can run it as a continuous per-stage worker. Most systems already have the per-entity body; the migration is mechanical.

Microagents — the only place the model decides #

One question, one typed answer

A microagent answers one question via one typed tool call. I/O, shell, parsing, retries, validation, side-effects, and final string formatting stay out of the model — the calling system owns every save and transition. The convention is roughly 1:1 component↔microagent: a microagent's output is stored on a component, then fed to the next stage.

Maker ≠ Checker. When you add a verification or judge stage, the generator and each critic must be distinct subjects (and ideally distinct models) so nothing grades its own output. The framework's verify→judge loop is built on exactly this separation.

The full microagent contract ships with the agl-ai dependency — see node_modules/agl-ai/docs/MICROAGENT.md. The shape is Agent.factory({ model, system_prompt, output_tool }) → one bounded call returning a typed object.

Gate logic — the most safety-critical code #

Which entities, and when to abort

Gate logic decides which entities a system selects and when it aborts processing one. When it's wrong, an entity silently gets "stuck" — never selected, or always skipped. To make it a first-class, debuggable construct, every piece of gate logic is wrapped in the Gate marker.

import { Entity, Gate } from 'pipeline'

targets = await Entity.query activityId, (e) ->
  return false unless Gate 'captured',   e.workflow?._stage is 'captured'
  return false if     Gate 'in backoff', Entity.inBackoff e
  # …falls through → this entity matches (a predicate matches UNLESS it returns false)

Gate(cond) / Gate(label, cond) is a pass-through — it returns the boolean it wraps, so it drops around existing logic without refactoring. Because all gate logic flows through this one marker, three things become possible:

  • pipeline check flags gate-ish lines (the query predicate and any continue/break/return-false guards) that are not wrapped in Gate. A deliberate exception carries a trailing # gate:ignore.
  • pipeline docs extracts each label + condition verbatim into a Gate Logic table — deterministically, no LLM.
  • Runtime records each gate's pass/fail for the current entity; gateTrace(entityId) returns its recent evaluations — the "why is X stuck?" view.

The query cap is the framework's, not yours. Entity.query returns up to pipeline_width matches and sets _G.currentEntityId before each predicate so Gate() attributes correctly. The cap is a cross-cutting default — keep it out of your gate logic.

Activities — many pipelines, one project #

The multi-pipeline abstraction

An Activity is one named pipeline. A project may host one or many; they execute independently over their own entities, schema, and entity dir. Adding an activity is YAML + files, not code — declare a manifest, drop in the systems/agents/schema it references, and the registry wires it up at startup.

# notes/activity.yaml — the single declarative description of an activity
id:    notes                # stable id used everywhere (dirs, URLs, schema lookups)
name:  Notes
stages: [original, classified, published]   # decorative phase labels for dashboards
pipeline:                # ordered systems — run in this order every tick
  - ingestSystem
  - classifySystem
  - publishSystem
agents:                  # agents to import at startup (incl. microagents)
  - 01-classify

Two equivalent dual-mode layouts are supported and may even coexist during a migration:

Activity-first (recommended)

Everything for one activity grouped under <id>/ — activity.yaml, schema.yaml, systems/, db/. Scales nicely when activities are genuinely independent. Paths default off the directory, so a minimal manifest declares only id/name/stages/pipeline/agents.

Legacy central

Each kind namespaced by activity — activities/<id>.yaml, systems/<id>/, schema/<id>.yaml, db/entities/<id>/. Still fully supported; if present it takes precedence and the activity-first scan is additive.

A single-activity project needs neither: the framework synthesizes one default activity over flat db/, so the activity-scoping is invisible. Every Entity / World / component accessor takes the activityId first to keep the layers scoped.

await Activities.loadAll()        # call once at startup (the loop does this)
Activities.all()                # array of loaded activities
Activities.get 'notes'          # one activity (or the synthesized default)
Activities.entityDir 'notes'    # resolve its entity dir (the Entity hot path)

CoffeeScript dynamic-import gotcha. CoffeeScript treats import as a static keyword and won't parse await import(path). Use the JS backtick passthrough: mod = await `import(moduleUrl)`. The registry uses exactly this to load each system module by name.

Entity API #

Disk-authoritative persistence, static & Golang-style

Entity is a static singleton; every method takes activityId first, then the entity (or id). There is no per-entity object graph threaded through the loop.

World & query #

The cache layer (performance only)

The World is the in-memory cache; disk stays authoritative. Systems never touch it directly — they pull via Entity.query — but it's there for tooling.

import { World } from 'pipeline'   # or _G.World
W = World.for activityId                  # this activity's cache handle
W.Entity__find (e) -> e.workflow?._stage is 'captured'   # the core query primitive
W.get id    ·   W.all()    ·   W.count()

defineComponent #

Generated, validated accessors

defineComponent(name, fields) generates a validated getter (camelCase(field)) and setter (set + PascalCase(field)) per field. Each accessor calls SchemaValidator.check with the calling subject before touching disk — so the ACL is enforced on every read and write.

export ReviewComponent = defineComponent 'review', ['approved_by', 'approved_at']

approver = await ReviewComponent.approvedBy SUBJECT, activityId, id        # getter
await ReviewComponent.setApprovedAt SUBJECT, activityId, id, isoNow            # setter

For multi-field or array writes, subclass Component and call @check(subject, field) per field — then list the method→fields mapping in schema-aliases.yaml so pipeline check can trace the compound write.

SchemaValidator #

The runtime ACL enforcer

import { SchemaValidator } from 'pipeline'
SchemaValidator.check subject, component, field   # fatal (exit 1) if the subject isn't allow-listed
SchemaValidator.checkScalar subject, scalarName    # same, for top-level scalar fields

It reads schema.yaml fresh on every call (no caching), so allowlist edits hot-reload with no restart. A component declared in multiple schema files has its subjects[] allowlists unioned.

pipeline check — the static linter #

The build fails before you ever run

The static analyzer validates the whole contract at once:

  • Every Component.method SUBJECT, … call site is checked against the schema allowlists; dangling (unreferenced) fields are reported.
  • Gate wrapping — gate-ish lines must use Gate(...) (a deliberate exception carries # gate:ignore).
  • Field-index — every gate-read field must be in the system's GATE_FIELDS (coverage) and be a real schema field (existence); over-declaration warns.

Project-specific compound accessors are declared in a project-root schema-aliases.yaml. Keep pipeline check green — it is the contract's CI gate.

pipeline review — LLM convention rules #

Judgment a compiler can't express

Where pipeline check enforces the mechanical schema/ACL contract, pipeline review runs bundled code-review rules — the subjective conventions an AST can't easily express (e.g. "every entity write must go through the transaction API"). Use both as the green bar before merging.

The ratchet. When you discover an invariant the codebase should always hold: (1) write a code-review rule for it, (2) run it and confirm the current offenders go RED, (3) fix until GREEN, (4) commit the rule with the change. The invariant is now enforced forever — every hardening buys a permanent, observable check.

pipeline docs — docs that can't drift #

Generated from the source of truth

pipeline docs regenerates the at-a-glance reference — pipeline → stage → system → component → field → subject, plus the Gate Logic table — directly from the schema and the Gate(...) markers. Because it is generated, the reference can't fall out of sync with the code.

The 14 invariants #

The contract every pipeline adheres to

These are the non-negotiables — enforced structurally by the SchemaValidator + pipeline check, and subjectively by pipeline review. If you only memorize one section before contributing, make it this one.

#Invariant
1Disk is authoritative; the World cache is only a performance layer. Crash and resume every entity where it left off. A chokidar watcher syncs operator edits back each tick.
2Systems pull, they are never pushed. A system's first act is Entity.query. Passing an entity/store into a system is an anti-pattern.
3Query by component presence (archetype-style). An entity is defined by which components it has; it accretes them as systems refine it.
4Static, Golang-style models. Entity/World are static singletons; methods take activityId first.
5Microagents do judgment only. I/O, parsing, retries, side-effects, final formatting stay out of the model. The calling system owns every save.
6Human gates are fields on disk, edited asynchronously. A null gate field = no-op this entity; a truthy value = advance. A gate never blocks the loop for other entities.
7Machine vs human fields by convention. A leading _ means machine-managed; an unprefixed field is the operator's.
8Stage lives inside the entity; files never move. Either workflow._stage or status + component-presence — Entity.query is agnostic.
9Systems run in configurable order, with bounded parallelism (pipeline_width) and per-entity retry/backoff.
10Schema is the single source of truth + a per-field ACL. Every field declares a subjects: allowlist; the validator is fatal on violation.
11Re-queue by component-deletion. Entity.drop(...) the keys; the archetype query re-picks the entity at the earliest missing stage. The basis of verify→refine loops.
12Maker ≠ Checker. The generator and each critic are distinct subjects (ideally distinct models) so nothing grades its own output.
13Gate logic goes through the Gate(...) marker — enforceable, debuggable, self-documenting.
14Field-index: selection runs on a projection; bodies hydrate on match. (Opt-in low-memory selection at scale — see below.)

Field-index — low-memory selection at scale #

How 1,000+ multi-megabyte entities stay in RAM

War story. The original loop bulk-loaded all 1,015 entities — 2.9 GB of YAML, individual files up to 33 MB — into the World cache, ballooning RSS to ~20 GB. The fix wasn't "load less data," it was "load only the fields the gates actually read."

When _G.useFieldIndex is on and an activity declares an index — the union of its systems' exported GATE_FIELDS — the World holds only a thin projection of each entity (its id + those dot-paths) instead of the full body. The mechanics:

  • Declare per system: export GATE_FIELDS = ['fetch.status', 'verify.pass', …] — the <component>.<field> paths its gate reads. The activity's index is the union across its pipeline.
  • Scan on projections. The gate predicate reads the cheap resident projection — fast, no disk.
  • Hydrate on match. Entity.query then loads each match's full body from disk before handing it to the system fn.
  • Re-project at the boundary. evictHydrated drops the hydrated set back to projections — the stage-boundary GC (per-entity evictHydratedOne for the worker pools).

The safety property. Entity.load returns the projection only while a gate is scanning (tracked per async-context). Every other load — and therefore every write path — sees a full body, so projections never truncate on save. Activities that declare no GATE_FIELDS stay in full-body mode (back-compat). pipeline check enforces coverage + existence.

CAS — the lost-update fix #

Optimistic concurrency on a cheap fingerprint

War story. Under wide concurrency, fully-processed entities were silently rewound to an earlier stage and the settled count shrank. The cause: every component setter did load → assign → whole-file save, so two concurrent setter chains on the same entity clobbered each other — the second save built from a stale base.

The fix is optimistic concurrency control on a cheap mtimeMs:size fingerprint:

  • Entity.begin / commit / rollback — a transaction whose state rides on entity._txn; commit() re-stats the file and, on a fingerprint mismatch, returns a conflict with no write.
  • Entity.mutate(id, fn) — the convenience wrapper: begin → fn → commit, auto-retrying on conflict (bounded). This is the blessed write path; the convenience writers (patch/merge/append/setPath/drop/transition) all route through it.
  • Writes are atomic (temp file + rename), which also fixes crash-truncation.
  • Entity.create is a no-op when the file exists — killing the re-create rewind path independent of cache correctness.

The convention rule. A code-review rule (pipeline-persistence-via-transaction) flags any setter that slips back to a raw Entity.save — so the lost-update class can never silently return. The rule is the spec and the regression test.

Worker-pool stages — per-stage concurrency #

A slow stage no longer blocks the loop

War story. The original loop swept stages serially to completion every tick. One slow stage (a 1–3 min browser render) head-of-line-blocked the entire loop — resolve, publish, and report sat idle while verify ground through its queue.

Now every stage runs as its own continuous worker pool, all concurrent. pipeline_width is the max concurrency per stage: at width 3, up to 3 resolve and 3 publish and 3 verify run at once. Each worker polls its gate, claims up to width − inflight eligible entities (excluding ones it's already processing), runs each, and refills a slot the instant one finishes.

A system opts in by exporting the two seams alongside its *System fn (which stays for pipeline walk):

export selectEligible = (activityId, { exclude, limit }) ->   # the gate, in-flight-excluded
  matches = await Entity.query activityId, GATE, { exclude, limit, order: 'mtime' }
  (String(e.id) for e in matches)

export processOne = (activityId, id) -> await doOneEntity id   # process exactly one
export onTimeout = (activityId, id) -> …   # optional: mark a wedged entity terminal
export cleanup   = -> …                    # optional: close a pool once at drain

Safety valves. A per-stage stage_timeout_ms aborts a pathological entity (e.g. verify past 5 min), marks it terminal so its slot frees, and the stage keeps flowing. Graceful shutdown drains in-flight entities (a second Ctrl-C force-quits). And because writes are CAS-safe, concurrent stages writing the corpus can't lose updates. A legacy fn-only system still runs — wrapped as a concurrent batch loop.

War stories — the "why" behind the design #

Every guardrail earned its place

This framework was forged migrating ~1,000 Kibana dashboards. Each invariant traces to a real failure:

🌩️ Infra ≠ defect

A flaky cluster intermittently 404s with node_disconnected even when _remote/info says "connected." A verify stage classifies these as infra and leaves the entity stuck without a refine — never thrash the corpus during an outage. That distinction is why the classification exists.

📌 Cluster-pin invariant

A data-view title that pins a literal cluster (tic-…-prod-03:…) errors the whole panel when that one node blips. The cluster segment must be a wildcard * — the resolver blocks a pinned title and rewrites it on copy.

🔁 Idempotent by title, not id

Kibana's _import remaps saved-object ids across spaces. So copy-to-scratchpad must be idempotent by title — and re-importing a shared firehose data view per-entity once produced an ambiguous_conflict that stranded a whole class until find-or-reuse fixed it.

🧪 Couldn't grade itself

An A/B vision judge once flipped its verdict between runs because it misread a mid-load spinner in the source screenshot as "had data." The fix taught the judge that a loading spinner is not evidence — the same Maker≠Checker discipline, applied to perception.

The throughline. Every one of these is the same lesson: when a deterministic step can't decide, it must stay stuck and self-explain — never guess, never thrash. The loop makes that cheap (the bad item waits; everything else flows), and the guardrails make it observable (gateTrace, block_reason, the generated docs).
